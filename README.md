# コンテンツ差別化戦略の検証基盤（PostgreSQL × YouTube Data API v3）

> **⚠️ 現在進行中のプロジェクト（データ収集中）**
>
> このプロジェクトは、その過程でサンプルサイズ不足による信頼性の低いインサイトが得られたため、YouTube Data API v3 のクォータ制限の範囲内で収集クエリを大幅に増やし、数万件規模の生データを複数日かけて再収集中である。
> YouTube Data API v3 の無料クォータ（1日 10,000 ユニット）の制約上、全クエリの実行完了まで**1〜2週間程度**かかる見込みで、
> 毎日クォータリセット後に収集スクリプトを再実行し、進捗ファイル（`progress.json`）で中断・再開を管理している。
> データ収集完了後、`## 4.` のインサイトセクションを最新データで更新予定。
> 下記の内容は、再集計前に作成した暫時的な記録であることに留意されたい。

## 1. このプロジェクトの目的

> 「自分が取った差別化戦略は、市場の中で本当に機能したのか？」

本プロジェクトは、この問いに答えるための**分析基盤（DWH）を設計・構築すること**を目的とする。
「分析できる状態を作り、分析者にバトンを渡す」ことがゴールであり、本格的な分析はスコープ外とする。

## 2. ビジネス背景と差別化戦略

### 市場の構造（仮説）

特定の動画投稿ジャンル（ゲーム関連）において、市場は以下の2軸で整理できる：

| 軸 | 支配的な戦略（市場の主流） | 本プロジェクトの主体が取った戦略 |
|---|---|---|
| プレイスタイルA | `[主流戦略]` が主流 | `[差別化戦略]` で対抗 |
| 映像要素B | `[主流戦略]` では演出がボトルネック化 | そのボトルネックを構造的に解消 |

### 私がとった差別化戦略

市場の主流戦略には、**映像演出面でのボトルネック**が存在していた。
具体的には、主流スタイルでは「プレイ画面は派手だが、`[映像要素]`が地味になる」という矛盾が生じ、
`[映像要素]`という本来の武器が逆に弱点になる構造があった。

私はこれを逆手に取り：

- `[差別化戦略]` であることで`[映像要素]`も派手になる
- プレイ画面の見栄えも両立できる
- `[差別化戦略]` × `[プレイスタイルA]` というニッチな組み合わせを、さらに`[差別化アトリビュート]`を**常識外れ**のレベルで尖らせ、**ブランディング**として機能させる

（実際に、`[差別化アトリビュート]` がどの程度「常識外れ」かを定量化するため、世界各国のプロ選手 157 名の設定と比較した結果、
**最も極端な設定**を扱っていることがわかっている。）

---


**プレースホルダについて**:
個人チャンネルの固有キーワードにつき具体名は伏せているが、上表の `[主流戦略]` / `[差別化戦略]` はそれぞれ本プロジェクトの
`mainstream` / `own_strategy` セグメントに対応する。

- `mainstream` <-- 市場の主流戦略をとるクリエイターの動画（収集クエリに `{GAME} {MAINSTREAM_ATTRIBUTE} {SKILL_KEYWORD}` ~~`{FOOTAGE_KEYWORD}`~~ を含むもの）  

- `own_strategy` <-- 自チャンネルと同じ差別化戦略をとるクリエイターの動画（収集クエリに `{GAME} {OWN_ATTRIBUTE} {SKILL_KEYWORD}` ~~`{FOOTAGE_KEYWORD}`~~ を含むもの）  

- `{GAME}` は特定のゲームタイトルを表すキーワード
- `{MAINSTREAM_ATTRIBUTE}` は市場の主流ゲーム設定のキーワード
- `{OWN_ATTRIBUTE}` は主流とは対をなす自チャンネルのゲーム設定キーワード
- `{SKILL_KEYWORD}` は特定のプレイスタイルを表すキーワード
- ~~`{FOOTAGE_KEYWORD}` は映像要素を表すキーワード~~

> ~~`{FOOTAGE_KEYWORD}`~~ について：YouTube Data API の検索は関連度スコアベースのため、このキーワードを収集クエリに含めても意図した手元映像動画を捉えられないことが判明し廃止。詳細は [仮説③](#仮説主流戦略のボトルネック構造の確認) 参照。


---

### 検証したい仮説

1. **差別化の実現確認** — 同じ方向性のクリエイターは市場に本当に少ないのか（ニッチ市場の選定ができているか）
2. **パフォーマンス比較** — 主流・差別化アプローチをとるクリエイターと比較し、本当に自分のコンテンツの方がハイパフォーマンスか（ブランディング効果の確認）
3. **仮説の整合性確認** — 【`[主流戦略]` × `[映像要素]`あり】と【`[差別化戦略]` × `[映像要素]`あり】 のパフォーマンスと比較し、ボトルネック構造（仮説）の整合性を確認

--> これらを検証するためのデータ基盤を構築し、分析者に渡すことがこのプロジェクトの仮想目的とする。

## 3. プロジェクト全体像

```
【データソース】
自分のYouTube Analytics CSV（自己データ）
YouTube Data API v3（競合・市場データ 約1,200本）*1

         ↓

【DWH: 3層構造】
raw 層   ← データをそのまま保存
clean 層 ← 型変換・重複排除・正規化
mart 層  ← 分析・比較できる形に統合

         ↓

【バトン先（スコープ外）】
データサイエンティスト / アナリストによる
差別化戦略の定量検証・可視化
```

## 4. 基盤構築後のインサイトと戦略仮説との照合

>このセクションは現在のデータで一度出力したインサイトだが、サンプルサイズが不十分なため信頼性が低い。
>理由: YouTube Data API の無料クォータ（1日 10,000 ユニット）の制限で1日に実行できるクエリ数が限られており、
>当初の収集データ（約1,200本）ではown_strategy・mainstreamそれぞれ数十〜百数十本程度しか集まらなかった。
>YouTube Data API v3 のサーチは関連度スコアベースであり厳密なフィルタリングができないため、
>実際に条件に当てはまる動画がさらに少なくなってしまい、プロジェクトが根本的に破綻する事態に陥った。

## 5. 制約事項

本プロジェクトには以下の制約があることを明記する：

| 制約 | 内容 |
|---|---|
| データ規模 | 自己データのサンプルサイズが小さく、統計的検出力は限定的 |
| 静的データ | 自チャンネルの投稿は1年以上停止しており、視聴回数等の指標は日々変動するが、本プロジェクトで使用しているのは収集時点のスナップショットである |
| 分析スコープ | 本プロジェクトはDWH基盤の構築までを担当し、統計分析・MLはスコープ外 |
| 正規化の仮定 | `normalized_avg_view_count` は登録者数と再生数の線形比例を仮定した参考値であり、知名度・コンテンツ品質・アルゴリズムなど非線形な要因は考慮していない |
| プロモーション動画 | 集計したコンテンツのプロモーションの有無に応じた補正はされていない（YouTube Data API v3 の制約でもある） |
| [映像要素]検出の限界 | タイトルキーワード判定では[映像要素]を含む動画の大半を検出できないため、動画・サムネイルのML解析が必要であり本プロジェクトのスコープ外 |
| プラットフォーム | X・TikTokはAPI制限のためスコープ外（スキーマ設計のみ考慮） |

以上の制約を踏まえた上で、セクション10の照合結果は「仮説と整合する傾向が見られる」という示唆に留まり、
定量的な検証は分析者に委ねる。本プロジェクトは**「実運用規模のデータが継続的に供給される前提で設計された基盤」** として位置づける。

## 6. 将来的な拡張

- **X / TikTok への対応** — APIが利用可能になった段階でソースを追加（スキーマは拡張想定済み）
- **クラウドDWHへの移植** — BigQuery / Snowflake へのSQL移植
- **リアルタイムパイプライン** — YouTube Analytics API による日次自動取得
- **分析レイヤー追加** — データサイエンティストによる差別化戦略の定量検証
- **因果分析** — プロモーション適用動画の効果補正、アルゴリズム要因の分離など、再生数の因果構造の解明
- **ML による[映像要素]検出** — サムネイルや動画フレームの画像解析により、タイトルキーワードに依存しない[映像要素]有無の検出を実現

また、このプロジェクトの raw → clean → mart の3層構造 は、Databricksで主流の Delta Lake
アーキテクチャ（Bronze → Silver → Gold） とほぼ同じ思想なので、

| 今のプロジェクト | Databricks / Delta Lake |
|---|---|
| raw層 | Bronze層 |
| clean層 | Silver層 |
| mart層 | Gold層 |
| PostgreSQL | Delta Lake（クラウドストレージ上） |
| psqlで手動実行 | Databricks Workflows で自動スケジュール実行 |

Databricksの学習や次のプロジェクトでのクラウドDWH移行もスムーズに行えるよう、設計段階からこのアーキテクチャを意識している。

## 11. 環境構築・再現手順

```bash
# 1. PostgreSQL を起動
docker compose up -d

# 2. テーブル作成
docker exec -i youtube_dwh_db psql -U yuji -d youtube_dwh_db -f /sql/01_create_tables.sql

# 3. 自己データをインポート・clean層変換
docker exec -i youtube_dwh_db psql -U yuji -d youtube_dwh_db -f /sql/02_import_and_clean_self.sql

# 4. 競合データを収集（YouTube Data API が必要）
python src/python/collect_competitor.py

# 5. 競合データをインポート・clean層変換
docker exec -i youtube_dwh_db psql -U yuji -d youtube_dwh_db -f /sql/03_import_and_clean_competitor.sql

# 6. mart層を構築
docker exec -i youtube_dwh_db psql -U yuji -d youtube_dwh_db -f /sql/04_mart.sql

# 7. 品質チェック
docker exec -i youtube_dwh_db psql -U yuji -d youtube_dwh_db -f /sql/05_quality_checks.sql
```

--> 一発で全てのSQLを実行するコマンドは下記（ただし、途中で競合データの収集が必要なため、実行前に `collect_competitor.py` を走らせる必要がある点に注意）：

```bash
  docker exec -i youtube_dwh_db psql -U yuji -d youtube_dwh_db \
    -f /sql/01_create_tables.sql \
    -f /sql/02_import_and_clean_self.sql \
    -f /sql/03_import_and_clean_competitor.sql \
    -f /sql/04_mart.sql \
    -f /sql/05_quality_checks.sql
```
